# Harness Engineering：一种面向智能体的软件工程范式

## 引子

随着大模型逐步进入真实的软件开发流程，很多团队开始讨论一个问题：

**如果智能体已经可以写代码，那么软件工程接下来最重要的变化是什么？**

很多人第一反应会落在 prompt 上，或者落在“让 AI 多写一点代码”上。但如果真的观察一些先进团队的实践，会发现重点并不只是“生成代码”，而是：

**如何为智能体搭建一个能稳定工作的工程系统。**

这正是我理解的 `harness engineering`。

它不是某一个工具，也不是一段万能提示词，而是一种新的工程思路：  
通过规格、文档、测试、可观察性、自动化反馈和交付流程，把智能体从“代码生成器”变成“可以持续参与软件交付的工程执行者”。

## 什么是 Harness Engineering

如果要用一句话定义：

> Harness engineering 是一种面向智能体的软件工程范式：  
> 用规格定义目标，用仓库承载上下文，用工具和测试提供反馈，用自动化闭环驱动实现、验证、修复与交付。

这里的 `harness`，可以理解成“工作支架”或“执行框架”。

它的重点不是替代工程师做判断，而是让智能体在一个受控、可验证、可持续的环境里工作。

换句话说，它解决的不是：

- “AI 能不能写这段代码”

而是：

- “AI 能不能在一个真实工程里把事情做成”
- “AI 能不能自己验证结果”
- “AI 能不能在出错时继续修复”
- “AI 能不能参与部署、回滚和后续治理”

## 它和 Spec Coding 有什么关系

很多人会觉得 harness engineering 很像 spec-first、spec coding，或者像亚马逊式的 PRFAQ / spec 驱动开发。

这种感觉是对的，但它们不是同一个层级的东西。

### Spec coding 解决的问题

Spec coding 的核心是：

- 在编码前写清楚需求
- 讲清楚目标、边界、约束和验收标准
- 减少边写边猜

它非常重要，因为智能体和人一样，如果目标不清楚，就会出现方向漂移。

### Harness engineering 解决的问题

Harness engineering 更进一步，它不只关心“写什么”，还关心：

- 智能体如何理解上下文
- 智能体如何运行和验证应用
- 智能体如何获得失败反馈
- 智能体如何修复问题
- 智能体如何交付和持续维护

所以我更愿意这样理解两者关系：

```text
Spec coding = 任务定义层
Harness engineering = 智能体软件工程操作系统
```

换句话说：

**Spec-first 是 harness engineering 的一部分，但 harness engineering 比 spec-first 更完整。**

## 为什么它会变得重要

因为当智能体进入真实开发流程后，最大的瓶颈往往不再是“能不能生成第一版代码”，而是下面这些问题：

1. 它知不知道仓库应该先看哪里
2. 它能不能分辨哪些约束是硬约束
3. 它改完之后能不能自己验证
4. 它失败时能不能得到足够明确的反馈
5. 它能不能在不破坏系统的前提下继续修复
6. 它能不能参与持续交付，而不是只交一堆未验证的 patch

如果这些问题没有解决，智能体即使很会写代码，也很难真正融入工程系统。

所以 harness engineering 的本质，是把软件工程流程本身变成一个智能体可以运行的系统。

## Harness Engineering 的核心组成

在我看来，一套比较完整的 harness engineering 至少包含 6 个部分。

### 1. 用 Spec 定义目标和边界

这一步和 spec coding 一样重要。

必须先定义：

- 目标是什么
- 不做什么
- 成功标准是什么
- 哪些行为不能回归

如果没有这一步，智能体很容易生成“看起来很努力、其实偏题”的结果。

### 2. 用仓库承载长期上下文

不能把上下文只留在聊天窗口里。

一个适合智能体协作的仓库，应该有：

- 一个短小的 `AGENTS.md` 作为地图
- 分层的 `docs/`
- 产品、架构、验收、运维、技术债等结构化文档

这样智能体每次进场时，不是重新猜，而是能顺着仓库本身找到有效上下文。

### 3. 把约束变成可执行检查

“请保持代码整洁”没有用。

真正有用的是：

- lint
- 类型检查
- 测试
- 静态检查
- 验收脚本
- CI

也就是说，重要约束不能只停留在文字里，而是要变成程序可以执行的检查。

### 4. 让应用对智能体可读、可测、可观察

这一步非常关键。

如果智能体只能读代码，却不能：

- 驱动浏览器
- 看 DOM
- 看截图
- 看日志
- 看 trace
- 看构建结果

那它本质上还是“写代码的人”，不是“能验证软件的人”。

一旦这层能力补上，智能体就具备了类似工程师的“感官系统”。

### 5. 把监督和纠错编码进系统

当测试失败、构建失败、页面行为不对时，系统应该自动给出足够多的反馈证据。

例如：

- 哪个测试挂了
- 哪个断言失败
- 哪个页面元素没出现
- 哪张截图显示布局错了
- 哪段 trace 能重放用户路径

这样智能体就不需要等人来“解释发生了什么”，而是可以根据反馈继续修复。

### 6. 把交付和治理纳入闭环

一套成熟的 harness 不只是为了写出第一版功能，而是为了支持：

- 提交
- PR
- 评审
- 构建
- 部署
- 回滚
- 技术债治理

也就是说，它最终要服务的是“持续交付”，而不是“一次性生成代码”。

## 一个可以复用的工作流

如果把 harness engineering 抽象成一套通用流程，我会用下面这 7 步来描述：

```text
1. 写 spec
2. 更新仓库文档
3. 让智能体实现功能
4. 自动运行检查和测试
5. 根据失败反馈继续修复
6. 通过后进入 CI / 部署
7. 把这次经验重新沉淀回仓库
```

它的关键不是每一步都用 AI，而是每一步都为 AI 提供了可以继续前进的结构。

## 它和传统开发有什么不同

传统开发通常更像这样：

- 人定义需求
- 人写代码
- 人调试
- 人点页面
- 人发布

而 harness engineering 更像这样：

- 人定义目标和边界
- 智能体写代码
- 智能体跑测试
- 系统给出反馈
- 智能体根据反馈修复
- 人只在关键判断节点介入

这并不是“人退出开发”，而是“人从直接写代码，转向设计系统、定义规则和做关键决策”。

## 一个实际项目里的轻量落地

就算不是大公司，也可以把 harness engineering 用在小项目里。

例如一个简单的个人健康管理工具，也可以这样落地：

- `product-spec.md` 定义功能边界
- `acceptance.md` 定义不能回归的行为
- `AGENTS.md` 定义仓库地图
- 检查脚本验证文档和页面结构
- Playwright 负责浏览器级验收
- GitHub Actions 负责发布前校验
- 失败时保留截图、trace、视频和 HTML report

这样一来，项目虽然不大，但已经具备了面向智能体协作的最小执行系统。

## 这种范式的价值

我认为 harness engineering 最重要的价值，不是让 AI 多写几行代码，而是带来 4 个变化。

### 1. 从“生成代码”升级到“交付软件”

真正难的不是第一版代码，而是验证、修复和持续交付。

### 2. 从“靠聊天维持上下文”升级到“靠仓库维持上下文”

上下文一旦进入仓库，智能体和人都更容易接手和延续工作。

### 3. 从“人工验收”升级到“自动反馈闭环”

每次改动都先让系统跑一轮，能大幅减少低级回归。

### 4. 从“AI 辅助编码”升级到“AI 参与工程系统”

这才是它真正和普通 AI coding 不一样的地方。

## 它的边界在哪里

当然，harness engineering 也不是万能的。

它仍然依赖人来做这些事：

- 定义真正正确的产品方向
- 决定高风险方案
- 处理跨团队协调
- 对复杂质量问题做最终判断

所以它不是“无人软件工程”，而是：

**以人为主导、以智能体为执行者的软件工程新分工。**

## 结语

如果说 spec coding 回答的是：

**“我们应该让系统做什么？”**

那么 harness engineering 回答的是：

**“我们应该怎样搭一个系统，让智能体能持续把事情做成？”**

这也是为什么我认为它值得被当成一种独立范式来看待。

因为未来软件工程的竞争力，很可能不只是“谁会用 AI 写代码”，而是：

**谁更会为 AI 构建可执行、可验证、可交付的工程系统。**
