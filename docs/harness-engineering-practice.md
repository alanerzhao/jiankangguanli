# Harness Engineering Practice

## 背景

这次实践的起点不是“想把工程体系做大”，而是一次真实的产品体验回调。

在一次较大范围的 UI/UX 升级后，页面虽然功能更多、反馈更丰富，但实际使用体验变差了。项目先通过一次回滚恢复到了更稳的版本，然后围绕一个更实际的问题继续推进：

**怎样让后续的改动更稳、更可验证、更容易交给智能体和人一起维护。**

## 当时面对的问题

1. 需求和边界主要散落在聊天记录里，仓库本身缺少清晰地图。
2. 页面改动是否“真的可用”，更多依赖人工点一点，而不是稳定验收。
3. 发布流程能上线，但缺少“先检查再发布”的硬约束。
4. 这个项目已经开始接入 Supabase、复盘视图和趋势图，复杂度正在慢慢增加。

换句话说，问题不在于功能太少，而在于仓库还没有形成一个适合长期迭代的支撑结构。

## 这次实践做了什么

### 1. 先回到稳定产品面

- 回滚了难用的大版本 UI 升级，恢复到更可用的主界面。
- 保留了长期复盘、趋势图和自动云同步这些已经验证过价值的能力。

这一步很重要，因为 harness engineering 的前提不是“继续堆系统”，而是先确保产品基线是稳的。

## 2. 补最小可用的仓库地图

新增了：

- `AGENTS.md`
- `docs/index.md`
- `docs/product-spec.md`
- `docs/architecture.md`
- `docs/acceptance.md`
- `docs/operations.md`
- `docs/tech-debt.md`

这里遵循了一个原则：

**`AGENTS.md` 只做地图，不做百科全书。**

也就是说，它负责告诉智能体“先看哪里”，而不是塞满所有规则。真正的产品目标、架构边界、验收标准和运维流程被拆到 `docs/` 里，减少单一说明文件腐烂的风险。

## 3. 把“口头要求”变成可执行检查

新增了两类零依赖检查脚本：

- `scripts/check-docs.mjs`
- `scripts/check-static-site.mjs`

它们解决的是最基本但很常见的问题：

- 关键文档是否存在
- 地图文档是否互相链接
- 页面关键挂载点是否还在
- `config.js` 是否保留必须字段
- `supabase-setup.sql` 是否仍然包含核心表

然后把这些检查统一进：

```bash
npm run check
```

这意味着今后不再只是“建议检查”，而是有一个明确入口去验证仓库是否还在受控状态。

## 4. 把浏览器级验收接进来

只靠静态检查还不够，因为健康管理工具的核心风险在交互链路。

所以又补了 Playwright，并写了真实的浏览器验收：

### 当前覆盖的 3 条关键路径

1. 保存每日记录并刷新后仍存在
2. 切换复盘周期和趋势指标时，图表与摘要同步变化
3. 导出 JSON 后再导入，数据能恢复

这一步的价值不只是“多了测试”，而是把最常见、最容易在 UI 改动中悄悄坏掉的路径放进了自动守护范围。

## 5. 把检查接进发布流程

更新了 GitHub Actions：

- `Validate Static Site`
- `Deploy Static Site to GitHub Pages`

现在发布流程不再只是“推上去就上线”，而是：

1. 先跑语法检查
2. 再跑文档检查
3. 再跑静态站点冒烟检查
4. 再跑 Playwright 浏览器验收
5. 通过后才部署到 GitHub Pages

这就是这次实践里最核心的 harness：让反馈回路进入默认路径，而不是靠人额外记得做。

## 结果

这次实践之后，项目从“可用的静态网页”变成了一个更适合长期迭代的小型智能体友好仓库。

已经具备的能力包括：

- 有清晰的仓库地图
- 有结构化文档作为记录系统
- 有统一检查入口
- 有浏览器级关键路径验收
- 有发布前自动校验
- 有 GitHub Pages 持续部署

对这个项目来说，这些改动的意义是：

- 后面继续改功能时，更容易判断是不是破坏了核心路径
- 让 Codex 接手时，不需要过度依赖聊天上下文
- 让回滚、排查和后续扩展都更有边界

## 这次实践里的经验

### 经验 1：先回到稳定版本，再谈体系化

如果当前产品体验本身就不稳，继续在其上堆工程结构只会让问题更复杂。先回滚，再建设，是这次很正确的一步。

### 经验 2：文档要短、分层、能导航

最有用的不是一个超长说明书，而是一个能让人和智能体快速找到正确上下文的入口系统。

### 经验 3：最小自动化比大量原则更有效

“记得检查一下”没有用，`npm run check` 和 GitHub Actions 才有用。

### 经验 4：浏览器测试一定要覆盖真实高频路径

对于这个项目，保存记录、切换复盘、导入导出，比花哨的 UI 细节更值得先被自动化守护。

## 当前还没做完的部分

这次只是第一轮，不是终点。还没覆盖的重点包括：

- 云同步成功/失败与异常恢复路径
- 历史记录编辑和删除
- 更细的图表交互
- 年度复盘更完整的报告能力
- GitHub Actions 里 Node 20 兼容层的后续升级

## 可以复用的做法

如果以后再做别的个人工具项目，这次实践里最值得复用的是这套顺序：

1. 先让产品回到稳定可用版本
2. 写一个短 `AGENTS.md`
3. 把产品、架构、验收、运维拆进 `docs/`
4. 建统一检查命令
5. 补 2 到 3 条最关键的浏览器验收
6. 把检查接进发布流程

这比一开始就追求完整平台化，更适合小项目稳步前进。
